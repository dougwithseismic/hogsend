# Completeness critic

## Gaps

- IDENTITY STITCHING IS UNSPECIFIED AND EVERYTHING DEPENDS ON IT. All six agents assert 'first-touch cross-session stitch' but none says HOW: where the durable UID is minted (client JS on landing vs server-side redirect), how it persists across sessions/devices (1st-party cookie? localStorage? IP+fingerprint?), and how a later ORGANIC form-fill re-binds to an earlier anonymous FB-click-with-no-fill. This is the single load-bearing schema decision; you cannot design tables in week one without it.
- NO CANONICAL DATA MODEL WAS PRODUCED. The verbatim tracked-fields list exists (sosId/crmId/perspectiveId, fbclid, ad_id, dealValue, pipelineStage, utm_*, etc.) but no agent gave the relational schema: sessions/touchpoints table, deals table, conversion-point-definition table, and how the three external IDs (sosId canonical + crmId/perspectiveId aliases) join. Builder needs DDL day one.
- THE MULTI-MODEL ATTRIBUTION ENGINE — the user's stated CORE requirement — has ZERO concrete algorithm or data flow anywhere. Agents only INFER it 'slots into the touchpoints table.' Missing: how fractional credit (first/last/linear/time-decay/position/Markov/Shapley) is computed, and critically how the internal multi-model ledger (for reporting/budgeting) is DECOUPLED from the single value-based lead_sold event sent to Meta. This is inferred-only, not designed.
- CONVERSION-POINT + VALUE-EXTRACTION CONFIG per CRM is inferred as {campaignId, crmStage, event, valueField} but never verified. Half the CRMs put value in the webhook payload (GHL, Pipedrive) and half require a hydrate/GET fetch (HubSpot, Attio, Monday, Salesforce). The per-client stage-field mapping (keyed on pipelineId+stageId), poll cadence, and how value/currency is normalized are unspecified — this is the product spine.
- CAPI IDEMPOTENCY across the dual webhook+poll paths is a week-one blocker, not a nice-to-have: both paths detect the same stage change, so you need a deterministic event_id derivation (e.g. hash(dealId+toStage)) to stop lead_sold firing twice. No agent specified the key.
- LEGAL/CONSENT BASIS for stitching identities across sessions BEFORE consent + 29-day UID retention + CAPI value dispatch under UK GDPR/PECR is flagged by 3 agents and resolved by none. This gates whether the entire design is lawful; needs an answer before architecture locks.
- fbclid PERSISTENCE LAYER is unresolved: server-side capture at click time vs reconstructing fbc from a browser cookie. This forks the architecture (do you need an edge redirect/collector service?). fbc format fb.1.<ts>.<fbclid> requires the original click timestamp — where is it stored for a sale that closes weeks later?
- PAYACA (UK, payaca.com) API surface is entirely unknown (name-collision with a US payments gateway noted); one of the named CRM plugins cannot be built without partner onboarding.
- PERSPECTIVE WEBHOOK PAYLOAD SHAPE is unconfirmed — whether the UID + all click IDs (fbclid/gclid) arrive natively or must be injected as manual hidden fields client-side. Determines whether the form layer can even carry the identity you need.
- The 'under-reported while in beta' public /data-feed reconciliation is unspecified: is the leaderboard near-real-time from the same ledger or a delayed export? Matters if you ship the proof-feed as a differentiator.
