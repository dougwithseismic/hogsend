## 6.1 The loop: event in, journey runs, event out

Hogsend's whole model is one loop:

1. An event arrives — from your backend (`POST /v1/ingest`), from a webhook source, or
   from PostHog. The same server-side events you send to PostHog in Chapter 3 are the
   events you send here.
2. The engine stores it, then routes it to every journey **subscribed to that event**.
3. The journey runs — it can wait days, check history, branch, send email, message
   Discord/Telegram — and fires its own events back (`journey:completed`, `email.*`,
   etc.) so the loop is measurable in PostHog.

The key property: **a journey is durable TypeScript.** `await ctx.sleep(...)` is a
real two-day pause that survives deploys and restarts — not a cron job, not a flowchart
approximating control flow. You write normal code; the engine makes it durable.

### Why "durable" is the whole game

It's worth understanding what that buys you, because it's the reason a code-first journey
engine exists at all. A lifecycle journey is mostly *waiting* — send, wait two days,
check, send, wait five days, check. Naively, you might reach for `setTimeout` or a
background `sleep`. That works until your server restarts mid-wait (a deploy, a crash, an
OOM) — and then every in-flight wait is lost, and every user mid-journey is stranded. You
can't run a messaging program on a process that forgets what it was doing every time you
ship.

Durable execution solves this: the engine persists the journey's position, so a
two-day sleep genuinely survives a redeploy on day one and resumes on day two as if
nothing happened. That's what lets you write a multi-day program as a single, readable
function instead of a pile of cron jobs and database flags reconstructing the state by
hand. The alternatives (§6.10) all exist to approximate this; Hogsend gives it to you as
plain control flow.

There's a subtlety inside "durable" that §6.9 returns to: durability is achieved by
being able to *replay* a journey from the top after a crash and fast-forward through
the parts that already ran. That replay model is why exactly-once sending needs engine
support rather than hope — hold the thought.

### Send the trigger events

From your backend, the same place you call `posthog.capture(...)`, also post to Hogsend:

```ts
await fetch(`${HOGSEND_URL}/v1/ingest`, {
  method: "POST",
  headers: { "content-type": "application/json", authorization: `Bearer ${KEY}` },
  body: JSON.stringify({
    event: "user.created",
    userId: "user_123",
    userEmail: "sam@example.com",
    properties: { plan: "free" },
  }),
});
```

That `user.created` event is what triggers the welcome journey below. One server-side
emission, two consumers: PostHog measures it, Hogsend acts on it. And because it's
server-side, it's ad-blocker-proof (Chapter 3) — your lifecycle program never depends
on a browser script that a third of your users block.

## 6.2 Anatomy of a journey

Every journey is `defineJourney({ meta, run })`. The `meta` declares *when* it runs and
the guardrails; `run` is the actual control flow.

```ts
import { days, hours } from "@hogsend/core";
import { defineJourney, sendEmail } from "@hogsend/engine";
import { Events, Templates } from "./constants/index.js";

export const example = defineJourney({
  meta: {
    id: "example",
    name: "Example journey",
    enabled: true,
    trigger: { event: Events.USER_CREATED },     // the event that starts it
    entryLimit: "once",                            // once | once_per_period | unlimited
    suppress: hours(12),                           // ignore re-triggers within this window
    exitOn: [{ event: Events.USER_DELETED }],      // events that end the journey early
  },
  run: async (user, ctx) => {
    // user: { id, email, stateId, journeyName, properties }
    // ctx:  durable primitives — sleep, history, waitForEvent, trigger, guard
  },
});
```

The pieces that matter for lifecycle work:

- **`trigger.event`** — the event the journey subscribes to. Add **`where`** to filter
  on properties: `trigger: { event: Events.USER_CREATED, where: (b) => b.prop("plan").eq("free") }`.
- **`entryLimit`** — `"once"` (a user can only ever enter once), `"once_per_period"`
  (with `entryPeriod`), or `"unlimited"`. This is your re-enrollment policy and an
  important correctness control.
- **`exitOn`** — the events that stop the journey immediately. **This is how you
  implement suppression from Chapter 5:** exit the upgrade sequence the instant the
  user upgrades. An exit fires even mid-wait — a journey parked on a 5-day
  `waitForEvent` aborts cleanly the moment an exit event arrives, so no post-wait
  side effects run.
- **`suppress`** — a debounce so a burst of the same trigger doesn't start multiple
  runs.

Inside `run`, you use the context primitives:

- `ctx.sleep({ duration, label })` — durable pause.
- `ctx.sleepUntil(at, { label })` — durable pause until an absolute instant.
- `ctx.when` — a timezone-aware scheduler for computing that instant: e.g.
  `ctx.when.tomorrow().at("09:00")` resolves 9am *in the user's own timezone* and
  respects your configured send window, so "send the day-2 email at breakfast, not at
  3am" is one line instead of timezone arithmetic.
- `ctx.history.hasEvent({ userId, event, within? })` → `{ found, count }` — did this
  already happen?
- `ctx.history.events({ userId, event, within, limit })` — pull the matching events to
  inspect.
- `ctx.waitForEvent({ event, timeout, label? })` — pause until *this user* emits an
  event or the timeout elapses; returns `{ timedOut, properties? }` (the matched
  event's payload, so you can branch on the answer). An optional `lookback` first
  checks recent history, covering the gap where the event fired a moment *before* the
  wait was established.
- `ctx.trigger({ event, userId, properties? })` — emit an event back into the pipeline.
- `ctx.guard.isSubscribed()` — re-check subscription after a long sleep.
- `ctx.checkpoint(label)` — mark progress so operators can see where users sit.

And you send with `sendEmail({ to, userId, journeyStateId, template, subject, props? })`,
where `template` is a key from your `src/emails/` registry (next section).

The thing to notice across all of this: it's *just code*. The branching is `if`. The wait
is `await`. The history check is a function call. There's no DSL to learn and no canvas
to fight — which is exactly why a journey can express logic a no-code tool can't, and why
it's reviewable in a pull request like the rest of your product.

## 6.3 Templates: email as code

The journeys decide *when* and *who*; templates decide *what lands in the inbox*. In a
Hogsend app, templates are **React components** — the
[React Email](https://react.email/docs/introduction) model, where an email is a
component tree (`<Container>`, `<Button>`, `<Text>`) that renders to the
table-based HTML that email clients actually tolerate. If you've ever hand-written
`<td>` soup for Outlook, this is the upgrade: components, props, type-checked data, and
your design system's tokens — compiled to HTML that works in clients stuck in 2009.

A template in your app is a component plus a registry entry:

```tsx
// src/emails/activation-welcome.tsx
export function ActivationWelcome({ firstName, ctaUrl }: Props) {
  return (
    <EmailLayout preview="Let's get you set up">
      <Text>Welcome{firstName ? `, ${firstName}` : ""} 👋</Text>
      <Text>One thing to do today: create your first project.</Text>
      <EmailButton href={ctaUrl}>Create a project</EmailButton>
    </EmailLayout>
  );
}
```

The engine owns the pipeline from there: when a journey calls `sendEmail({ template:
Templates.ACTIVATION_WELCOME, props })`, the engine renders the component to HTML,
checks the user's preferences/unsubscribe state, rewrites links for first-party click
tracking, records the send, and only then hands plain HTML to the delivery provider
(Resend by default — [their docs](https://resend.com/docs/introduction) cover the wire;
[Vercel's guide](https://vercel.com/kb/guide/sending-emails-from-an-application-on-vercel)
shows the same stack outside Hogsend). Two consequences worth noticing:

- **Templates version with your code.** The welcome email that shipped with v1.4 is in
  the v1.4 git history — reviewable, revertable, diffable
  ([Resend's templates post](https://resend.com/blog/introducing-templates) makes the
  same versioning argument even for their hosted flavour).
- **Content vs plumbing stays split.** Journeys never concatenate HTML; templates never
  decide who gets mail. The same separation your product keeps between logic and view.

## 6.4 Welcome + onboarding (Activation)

The first journey to build (Chapter 5, §5.3). Trigger on signup, welcome immediately,
then branch two days later on whether the user reached the aha moment:

```ts
import { days, hours } from "@hogsend/core";
import { defineJourney, sendEmail } from "@hogsend/engine";
import { Events, Templates } from "./constants/index.js";

export const activationWelcome = defineJourney({
  meta: {
    id: "activation-welcome",
    name: "Activation — Welcome Series",
    enabled: true,
    trigger: { event: Events.USER_CREATED },
    entryLimit: "once",
    suppress: hours(12),
    exitOn: [{ event: Events.USER_DELETED }],
  },

  run: async (user, ctx) => {
    await sendEmail({
      to: user.email,
      userId: user.id,
      journeyStateId: user.stateId,
      template: Templates.ACTIVATION_WELCOME,
      subject: "Welcome — let's get you set up",
      journeyName: user.journeyName,
    });

    await ctx.sleep({ duration: days(2), label: "post-welcome" });

    // Branch on behaviour, not on a guess (Chapter 5, §5.7).
    const { found: activated } = await ctx.history.hasEvent({
      userId: user.id,
      event: Events.FEATURE_USED,
    });

    await sendEmail({
      to: user.email,
      userId: user.id,
      journeyStateId: user.stateId,
      template: activated ? Templates.ACTIVATION_ADVANCED : Templates.ACTIVATION_NUDGE,
      subject: activated ? "Nice work — here's what to try next" : "You haven't tried the key feature yet",
      journeyName: user.journeyName,
    });
  },
});
```

Two things to notice. The journey **reads event history** to decide what to send — that's
behavioural personalisation in code, the §5.7 principle made literal: an activated user
gets "here's what's next," a stuck user gets "here's how to start," from the same journey.
And the whole thing **exits the moment `user.deleted` fires**, so a churned user never
gets the day-2 email. That `exitOn` is the difference between a system that feels
attentive and one that emails dead accounts.

One refinement worth adding once the basic journey works: replace the raw
`ctx.sleep(days(2))` with `ctx.sleepUntil(ctx.when.in(days(2)).at("09:00"))` so the
day-2 email lands at 9am in *the user's* timezone instead of exactly 48 hours after a
2am signup. Small change, visibly more human program.

## 6.5 A behavioural nudge series (Activation)

A multi-step version that escalates only when the user *hasn't* progressed, and exits
as soon as they activate. Note the second `exitOn` event:

```ts
export const activationNudgeSeries = defineJourney({
  meta: {
    id: "activation-nudge-series",
    name: "Activation — Behavioural Nudges",
    enabled: true,
    trigger: { event: Events.USER_CREATED },
    entryLimit: "once",
    suppress: hours(12),
    // Exit the instant they activate — no more nudges (suppression in action).
    exitOn: [{ event: Events.USER_DELETED }, { event: Events.USER_ACTIVATED }],
  },

  run: async (user, ctx) => {
    await ctx.sleep({ duration: days(2), label: "initial-nudge" });

    const { found: usedFeature } = await ctx.history.hasEvent({
      userId: user.id,
      event: Events.FEATURE_USED,
      within: days(2),
    });
    if (!usedFeature) {
      await sendEmail({
        to: user.email, userId: user.id, journeyStateId: user.stateId,
        template: Templates.ACTIVATION_NUDGE_SERIES,
        subject: "You haven't tried the key feature yet",
        journeyName: user.journeyName,
      });
    }

    await ctx.sleep({ duration: days(1), label: "setup-check" });

    const { found: setupDone } = await ctx.history.hasEvent({
      userId: user.id,
      event: Events.SETUP_COMPLETED,
      within: days(3),
    });
    if (!setupDone) {
      await sendEmail({
        to: user.email, userId: user.id, journeyStateId: user.stateId,
        template: Templates.ACTIVATION_QUICKSTART,
        subject: "Need help getting set up?",
        journeyName: user.journeyName,
      });
    }
  },
});
```

This is exactly the 3–5 touch behavioural sequence Chapter 5 recommended — except every
touch is gated on real behaviour, so an active user is left alone and a stuck user gets
help. The pattern generalises: *sleep, check what happened during the sleep, send only if
needed.* That single shape covers most onboarding programs you'll ever write.

Notice also what the `exitOn: USER_ACTIVATED` buys over the `if` checks alone: the
checks gate *sends at known points*; the exit kills the journey *at any point*,
including mid-sleep. Both layers matter. The `if` makes each message conditional; the
exit guarantees the whole program stops the moment its goal is met — Chapter 5's
suppression rule enforced by the engine rather than by your discipline.

## 6.6 Win-back (Retention)

Trigger on a dormancy event (you emit `user.dormant_30d` from a scheduled check, or via
a PostHog cohort webhook — see §6.11), then re-engage. Give the user a window to come
back on their own before sending again with `ctx.waitForEvent`:

```ts
export const reactivationWinback = defineJourney({
  meta: {
    id: "reactivation-winback",
    name: "Reactivation — Win-back",
    enabled: true,
    trigger: { event: Events.DORMANT_30D },
    entryLimit: "once_per_period",
    entryPeriod: days(30),
    exitOn: [{ event: Events.USER_DELETED }, { event: Events.APP_ACTIVE }],
  },

  run: async (user, ctx) => {
    await sendEmail({
      to: user.email, userId: user.id, journeyStateId: user.stateId,
      template: Templates.REACTIVATION_CHECKIN,
      subject: "We miss you — here's what's new",
      journeyName: user.journeyName,
    });

    // Give them 5 days to come back before the final nudge.
    const { timedOut } = await ctx.waitForEvent({
      event: Events.APP_ACTIVE,
      timeout: days(5),
      label: "comeback-window",
    });

    if (timedOut) {
      await sendEmail({
        to: user.email, userId: user.id, journeyStateId: user.stateId,
        template: Templates.REACTIVATION_FINAL_NUDGE,
        subject: "One last thing before you go",
        journeyName: user.journeyName,
      });
    }
  },
});
```

`ctx.waitForEvent` is the primitive that makes win-back feel human: instead of blindly
sending the second email on a timer, the journey *waits to see if they came back* and
only sends the final nudge if they didn't. And `exitOn: [{ event: Events.APP_ACTIVE }]`
means if they return at any point, the journey stops — they won't get a "we miss you"
email the day after they logged back in. Nothing is more corrosive to trust than a
win-back email to someone who's already active.

One habit for any journey with long waits: after the wait, re-check
`ctx.guard.isSubscribed()` before sending. An exit event ends a journey; an unsubscribe
doesn't — it's a preference the mailer honours at send time — but checking first keeps
your journey's *logic* honest about who still wants to hear from you.

## 6.7 Dunning (Revenue)

The highest dollar-ROI journey for a paid product (Chapter 5, §5.3). Trigger on a failed
charge, exit the moment payment succeeds:

```ts
export const dunning = defineJourney({
  meta: {
    id: "dunning",
    name: "Revenue — Failed Payment Recovery",
    enabled: true,
    trigger: { event: Events.PAYMENT_FAILED },
    entryLimit: "once_per_period",
    entryPeriod: days(14),
    exitOn: [{ event: Events.PAYMENT_SUCCEEDED }, { event: Events.SUBSCRIPTION_CANCELLED }],
  },

  run: async (user, ctx) => {
    await sendEmail({
      to: user.email, userId: user.id, journeyStateId: user.stateId,
      template: Templates.CHURN_PAYMENT_FAILED,
      subject: "Your payment didn't go through",
      journeyName: user.journeyName,
      props: { updateUrl: String(user.properties.updateUrl ?? "") },
    });

    // Wait 3 days for the retry (or the user) to fix it. If PAYMENT_SUCCEEDED
    // fires in the window, the wait resolves early — and the exitOn above ends
    // the journey anyway. Only the true timeout path sends the reminder.
    const { timedOut } = await ctx.waitForEvent({
      event: Events.PAYMENT_SUCCEEDED,
      timeout: days(3),
      label: "first-retry",
    });

    if (timedOut) {
      await sendEmail({
        to: user.email, userId: user.id, journeyStateId: user.stateId,
        template: Templates.CHURN_PAYMENT_FAILED,
        subject: "Reminder: update your payment method",
        journeyName: user.journeyName,
        props: { updateUrl: String(user.properties.updateUrl ?? "") },
        idempotencyLabel: "dunning-reminder",
      });
    }
  },
});
```

The `exitOn: PAYMENT_SUCCEEDED` is the whole point — the moment the retry clears (or the
user fixes their card), the reminders stop. This recovers a meaningful share of the
20–40% of churn that's involuntary (Chapter 5). The trigger comes from your payment
provider's failed-charge webhook (a Stripe `invoice.payment_failed` → `defineWebhookSource`
→ `payment.failed`), with the hosted card-update URL passed through as a prop so the
email has a one-click fix.

Note the `idempotencyLabel` on the second send: this journey sends the **same template
twice** on one run, so each send site declares a distinct label for the engine's
exactly-once bookkeeping — §6.9 explains the rule.

## 6.8 Referral (Referral)

Ask at peak satisfaction — trigger off a milestone, and only ask users who are still
active:

```ts
export const referralInvite = defineJourney({
  meta: {
    id: "referral-invite",
    name: "Referral — Post-Achievement Invite",
    enabled: true,
    trigger: { event: Events.MILESTONE_REACHED },
    entryLimit: "once_per_period",
    entryPeriod: days(7),
    suppress: days(2),
    exitOn: [{ event: Events.USER_DELETED }],
  },

  run: async (user, ctx) => {
    await ctx.sleep({ duration: hours(2), label: "post-achievement" });

    // Only ask people who are actually engaged.
    const { found: active30d } = await ctx.history.hasEvent({
      userId: user.id,
      event: Events.SESSION_COMPLETED,
      within: days(30),
    });
    if (!active30d) return;

    await sendEmail({
      to: user.email, userId: user.id, journeyStateId: user.stateId,
      template: Templates.RETENTION_ACHIEVEMENT,
      subject: "Share the love — invite a friend",
      journeyName: user.journeyName,
      props: { ctaText: "Invite a Friend" },
    });
  },
});
```

This encodes the Chapter 5 rule "ask at peak satisfaction" precisely: it triggers off a
*milestone*, waits a beat so the ask rides the good feeling, and the `if (!active30d)
return` guard means you never ask a disengaged user to vouch for you. The referral
*loop* — turning that invite into a tracked, rewarded conversion on an identity graph you
own — is Chapter 8. This journey is just the ask.

## 6.9 Correctness: exactly-once and re-enrollment

The failure mode that kills trust in a lifecycle program isn't a missed email — it's a
**double send**. Here's why it's a real risk and how the engine closes it.

Remember §6.1: durability works by *replaying* a crashed journey from the top and
fast-forwarding through completed steps. Now imagine a worker dies **after** the
provider accepted your welcome email but **before** the engine recorded that step as
complete. The replay re-runs the send line. Without engine support, that user gets two
welcomes — and at scale, every deploy is a small chance of embarrassing some slice of
your users. Three things worth internalising:

- **Replay safety is automatic.** The engine derives a deterministic idempotency key
  for every `sendEmail()` and `ctx.trigger()` from the run's identity plus the nearest
  authored wait label, and dedupes on it — so a replayed send collapses into the
  original instead of delivering twice. You write nothing extra.
- **The one authoring rule:** if you send the **same template twice in one journey**
  (like the dunning reminder in §6.7), give each site a distinct `idempotencyLabel` so
  their keys can't collide. Forgetting isn't silent — the engine throws a loud
  key-collision error in dev, which is exactly when you want to find out.
- **Keep the run deterministic where it matters.** If a *non-deterministic* choice
  picks what to send — an LLM choosing a template, a random variant — wrap it in
  `ctx.once(label, compute)`: the engine records the first result and replays it
  verbatim, so a crash can't re-roll the dice into a different-but-duplicate message.
  Same logic for timestamps: prefer the event's own `occurredAt` (from
  `ctx.waitForEvent`'s returned properties) over `new Date()`.

And separately from replays, **`entryLimit` is your re-enrollment policy** — it governs
legitimate *new* runs, not crashed ones. `"once"` for a welcome (a user signs up once).
`"once_per_period"` for win-back and dunning (they can lapse again, but not be spammed).
`"unlimited"` only for counting journeys (e.g. the Discord gamification in Chapter 8,
where every reaction must re-count).

This is the kind of correctness you get *because* the journey is code in a system built
for it — exactly-once delivery across redeploys is brutally hard to get right by hand,
and it's the subtlest of the reasons a purpose-built engine beats the roll-your-own
options in the next section.

## 6.10 Other ways you could build this (and why durable execution)

A growth team should know what it's choosing instead of. The ways teams build lifecycle
automation, roughly worst-to-best for this job:

- **`setTimeout` / in-process timers.** The tempting first reach, and the one that
  hurts: every wait is lost on the next deploy or crash. Fine for a 30-second
  debounce, catastrophic for a 3-day sequence.
- **Cron + a state column.** A nightly job scans the users table and decides who's due
  for which email. It works, and plenty of companies run on it — but you end up
  hand-rebuilding state, idempotency, and branching that the engine gives you for free,
  and the logic smears across cron jobs and SQL until no one can read the program.
  The tell: six months in, nobody can answer "what messages can user X possibly
  receive this week?" without reading four jobs and a spreadsheet.
- **A job queue with delayed jobs (BullMQ, Sidekiq).** Better — delays survive restarts.
  But you're still assembling sequences, exit conditions, and exactly-once yourself, job
  by job. Each journey becomes a distributed state machine you maintain by hand.
- **Durable-execution platforms (Temporal, Inngest, DBOS, Restate).** The right
  *category* — they make multi-step, long-running, restart-proof workflows a first-class
  thing. Hogsend uses one (Hatchet) under the hood. If you're building generic
  workflows, these are excellent; for lifecycle *messaging* specifically you'd still
  build the email/preferences/tracking/identity layer on top, which is what Hogsend is.
- **No-code automation (Zapier, n8n, Make).** Great glue for "when X, do Y" one-shots;
  weak for stateful multi-day per-user journeys with branching. Useful at the edges.

The summary: lifecycle messaging is a *durable, stateful, per-user* problem, and the
tools that fit it are the ones that treat long waits and exactly-once delivery as solved.
Hogsend's bet is that giving you that as ordinary TypeScript — plus the email, identity,
and tracking layers already wired — beats both the roll-your-own stack and the no-code
canvas for a team that codes.

## 6.11 Where the triggers come from

Your journeys are only as good as the events feeding them. Three sources:

- **Your backend** — `POST /v1/ingest` for product events (signup, payment, milestone).
  This is the primary source and it's server-side, so ad-blocker-proof (Chapter 3).
- **Webhook sources** — `defineWebhookSource()` turns any external webhook (Stripe,
  a form, a CRM) into an event that can trigger a journey. Declare the auth header,
  optionally validate the payload, and write one `transform()` that maps the
  provider's shape onto your event names — from then on a Stripe invoice failure is
  just another event in the same pipeline.
- **PostHog** — for *batch* / analytics-defined audiences (e.g. "did signup but not
  activate in 7 days"), build a PostHog cohort + a workflow webhook that posts to
  `/v1/ingest`. PostHog decides *who* needs action; Hogsend decides *what* to do. This
  is the bridge that powers inactivity-based win-back without Hogsend polling anything.

That last one is the clean expression of "PostHog detects, Hogsend acts" (Chapter 0):
the thing that's hard to compute in real time (30-day dormancy across all users) is a
cohort PostHog already maintains; Hogsend just receives the verdict and runs the program.

---

## 6.12 The holdout hook

Chapter 5 said: prove lift with a holdout. Because enrollment is just code, you can
implement a campaign holdout directly — deterministically bucket a small fraction of
users out of enrollment and compare their downstream events to the enrolled group:

```ts
run: async (user, ctx) => {
  // ~10% holdout, deterministic per user so it's stable across replays.
  if (hashToUnitInterval(user.id) < 0.1) {
    await ctx.trigger({ event: "journey.held_out", userId: user.id });
    return; // measured, not messaged
  }
  // ...normal journey...
}
```

Two details are load-bearing. The bucketing is a **hash of the user id**, not
`Math.random()` — deterministic, so a replay makes the same decision and the same user
is always in the same bucket across every journey version. And the held-out path still
**fires an event** — the control group must be *visible* in PostHog, or you'll have
nothing to compare against.

Then in PostHog, compare activation/retention/revenue for `journey.held_out` users vs
enrolled users. That difference is your real, causal lift — the growth-team experiment
loop (Chapter 0) running continuously, in production, on the journey you just shipped.
When Chapter 9's monthly review asks "what's the lifecycle program actually worth?",
this hook is where the answer comes from.

## 6.13 Before you move on

The build order that tends to work, if you want a checklist for the week:

1. **Wire your backend** to `POST /v1/ingest` for `user.created`, `payment.failed`,
   and your milestone event — every journey below hangs off these three.
2. **Build `activation-welcome` first**; add its templates to `src/emails/` and
   preview them in the React Email dev server before they send (§6.3) — a broken
   layout is much cheaper to catch in the preview than in a customer's inbox.
3. **Add dunning if you charge**, then **win-back** — the same order as Chapter 5,
   for the same reasons.
4. **Put a holdout on that first journey** and watch the lift in PostHog — so when
   someone asks whether the program works, there's a number instead of a feeling.

You've now plugged the bucket. The rest of the course opens the tap — and makes sure
every drop lands in an audience you own.
