## 3.1 Ways to install

PostHog has an SDK for wherever your code runs. Pick by where the event happens, not by
habit — and most real products use two or three of these together.

**The browser snippet** — paste a `<script>` into your `<head>`. No build step, works on
any site, good for a marketing page or a quick start.

**Browser via npm** — `npm install posthog-js`. Better for a real app: types,
tree-shaking, and framework wrappers (`posthog-js/react`, Next.js helpers). Use this for
your product front-end.

```js
import posthog from "posthog-js";

posthog.init("phc_YOUR_PROJECT_KEY", {
  api_host: "https://us.i.posthog.com", // use eu.i.posthog.com on EU Cloud
  person_profiles: "identified_only",   // see §3.4 — this is the cost-sane default
});
```

**Server-side** — `posthog-node`, plus official libraries for Python, Ruby, Go, PHP,
Java, and more. This is where your *important* events should fire (§3.3), and it's the
half most beginner setups skip. A server SDK needs the user's id passed explicitly,
because there's no browser cookie to read it from.

**Mobile** — React Native, iOS (Swift), Android (Kotlin), and Flutter SDKs, with their
own autocapture and session-replay support. If you ship an app, instrument it with the
native SDK rather than trying to route everything through a webview.

That `phc_` key is your **project API key**. It's safe in client-side code: it can
*write* events but cannot *read* your data. (Reading data needs a separate personal API
key, which you keep secret.)

## 3.2 Autocapture: what you get for free

The moment `init()` runs, `posthog-js` starts recording a lot with no further code:

- **`$pageview`** and **`$pageleave`** — every navigation, with URL, referrer, and UTM
  parameters.
- **`$autocapture`** — clicks, input changes, and form submissions on `a`, `button`,
  `form`, `input`, `select`, `textarea`, and `label` elements.
- **`$rageclick`** — three-plus clicks in about a second (a frustration signal),
  plus dead clicks and copy/paste events.
- **Automatic properties** on everything — browser, OS, device, viewport, referrer,
  UTMs, and ad-click IDs like `gclid`/`fbclid` (these matter in Chapter 7).

Autocapture has sane privacy defaults: it records input *names* and *ids*, never input
*values*, and excludes password fields. You can suppress capture on any element by
giving it the class `ph-no-autocapture`.

Autocapture is your safety net — it means you're never *not* collecting data while you
decide what matters. Its limit is that it captures *interface* events ("a button was
clicked"), not *meaning* ("the user activated"). That gap is the next section, and it's
the difference between data and signal.

> **Trade-off worth naming:** autocapture is fast to start and great for exploration, but
> it can get noisy and, at high volume, expensive. The mature pattern is autocapture *on*
> for discovery, plus a deliberate set of named custom events (§3.5) for the funnels you
> actually report on. Some teams turn autocapture off in production once their custom
> events are solid; most leave it on for the replay/debugging value.

## 3.3 Custom events: the signal you must add yourself

Autocapture records that someone clicked *a button*. It does not know that the click
meant "user signed up" or "project created" — the events that define your funnels.
Those you capture explicitly.

In the browser:

```js
posthog.capture("user_signed_up", {
  plan: "free",
  signup_method: "email",
});
```

On the backend (with `posthog-node`), you must pass the user's id:

```js
client.capture({
  distinctId: "user_123",
  event: "subscription_started",
  properties: { plan: "pro", amount: 49 },
});
```

The rule that decides where to fire an event: **anything money- or
decision-critical** — signup, activation, subscription, purchase — **should be
captured from your backend, not the browser.** It's more reliable (no dropped requests)
and, as §3.7 explains, ad blockers cannot touch it. Capture UI interactions and
journey steps client-side, where some loss is acceptable.

This server-side rule is also the seam where Hogsend lives: the same backend events
that you send to PostHog can be sent to Hogsend's ingest endpoint to trigger lifecycle
journeys. One event, two consumers — PostHog measures it, Hogsend acts on it. Designing
your important events as *server-side* from the start means you get accurate analytics and
ad-blocker-proof journey triggers from the same line of code (Chapter 6).

## 3.4 Identifying users (and the cost lever hidden in it)

Out of the box, events are tied to an anonymous id (a random UUID the SDK stores in the
browser). To connect a person's activity to *who they are*, call `identify` — typically
right after login:

```js
posthog.identify("user_123", {     // your stable user id (DB id or email)
  email: "sam@example.com",
  name: "Sam Rivera",
});
```

This links all the prior anonymous activity from that browser to the identified person,
so the pre-signup and post-signup history becomes one timeline. Call it early and pass
the properties you know each time.

**Person properties** come in two flavours, and the difference matters:

- **`$set`** — overwrite every time. Use for things that change: email, plan, company.
- **`$set_once`** — write only if not already set. Use for immutable firsts: signup
  date, first referrer, acquisition channel.

```js
posthog.capture("user_signed_up", {
  $set: { plan: "free" },
  $set_once: { signup_date: "2026-06-29", initial_referrer: document.referrer },
});
```

Two things people forget:

- **Call `posthog.reset()` on logout.** Otherwise the next person on a shared machine
  gets merged into the previous user's profile.
- **`$set_once` checks whether the key exists, not which event is newest.** With
  out-of-order delivery, the first event *ingested* wins.

### The cost lever: anonymous vs identified

Anonymous events (no person profile) are roughly **4× cheaper** than identified ones.
The `person_profiles` setting controls this:

- **`identified_only`** (recommended) — events stay anonymous until you call
  `identify`. Logged-out marketing-site traffic costs the cheap rate; logged-in users
  get full profiles.
- **`always`** — every event creates a profile. More expensive; only if you genuinely
  need person data on everything.

For a typical SaaS, `identified_only` is correct and can cut your bill significantly.
This is the single biggest cost decision in the whole setup (Chapter 1, §1.5) — get it
right on day one.

### B2B: group analytics

If you sell to companies, you'll want to roll events up to the *account*, not just the
user, so you can ask "how many *companies* are active" and run account-level funnels:

```js
posthog.group("company", "company_456", { name: "Acme", plan: "enterprise" });
```

Two gotchas: unlike `identify`, you must send the group with **every** relevant event
or you lose the rollup — and group analytics is often a paid add-on with billing
implications (Chapter 1, §1.5). Check before you switch it on.

## 3.5 The events to track first

You do not need a hundred events. You need a small, deliberately named set that lets
you build the AARRR charts from Chapter 2. A good starter kit:

```
user_signed_up          # Acquisition  — fire server-side
user_logged_in          # Engagement
<your_activation_event>  # Activation   — the "aha" action, e.g. project_created
invite_sent             # Referral
subscription_started    # Revenue      — fire server-side
subscription_cancelled  # Revenue / churn
payment_succeeded       # Revenue      — fire server-side
<your_core_action>      # Retention    — the thing users do repeatedly
```

The one most teams forget is the explicit **activation event**. If you don't fire
`project_created` (or whatever your aha action is), you'll later try to reverse-engineer
it from autocaptured button clicks — and your activation funnel, retention cohorts, and
LTV analysis all break. Define it now (you found *which* action it should be in Chapter
2, §2.3).

### Naming: the rules that keep this usable

PostHog recommends an `object_action` shape with snake_case and present-tense verbs:

- ✅ `project_created`, `invite_sent`, `pricing_page_viewed`
- ❌ `ProjectCreated`, `user-signed-up`, `Sent Invite`

Properties follow `object_adjective`, with `is_`/`has_` for booleans and `_date`/
`_timestamp` for times: `item_price`, `is_subscribed`, `last_login_timestamp`.

**The single most important rule: never build event names dynamically.** This is the
mistake that ruins an analytics setup:

```js
// ❌ Don't — this creates thousands of un-queryable event types
posthog.capture(`page_viewed_${pageName}`);

// ✅ Do — one event, the variable goes in a property
posthog.capture("page_viewed", { page_name: pageName });
```

Aim for roughly **10–200 event types** total. Fewer and you can't see your funnels;
more and you've built a dictionary no one can read.

### The tracking plan: a growth team's most boring, most important document

The thing that separates a team's analytics from a mess is a **tracking plan** — a short,
shared spec of every event you fire, its properties, and where it fires. It's a Google
Sheet or a Markdown file in the repo, and it's owned, not vibes:

| Event | When it fires | Where | Key properties |
|---|---|---|---|
| `user_signed_up` | account created | server | `plan`, `signup_method`, `$set_once: signup_date` |
| `project_created` | first project saved | server | `template`, `is_first` |
| `subscription_started` | checkout success | server (Stripe webhook) | `plan`, `amount`, `interval` |

Why it earns its keep: it stops two engineers inventing `signup` and `user_signed_up` for
the same thing; it makes "events are part of done" (Chapter 0) enforceable in code review;
and it's the reference Chapter 6's journeys and Chapter 7's ad conversions both read from.
Tools exist for this at scale (Avo, Iteratively, RudderStack's tracking plans) but a
single shared doc beats nothing by a mile — start there.

## 3.6 Other ways to get events in

Capturing from your own code is the default and the cheapest. But it's worth knowing the
alternatives so you choose deliberately:

- **A CDP pipe (Segment / RudderStack).** Instrument once against the CDP's API; it fans
  the same events out to PostHog *and* your warehouse *and* your ad tools. **Worth it
  when** you have many destinations and want one canonical schema; **overkill when**
  PostHog is your only destination — then you're adding a hop and a bill for nothing.
- **Google Tag Manager.** Marketers love it because it adds tags without a deploy. The
  catch for a product team: GTM-injected tags are *more* likely to be ad-blocked (they
  load from Google's domains), and it pulls your event logic out of the codebase where it
  can be code-reviewed. **Fine for marketing-site pixels; not where your product funnel
  should live.**
- **Reverse-ETL / warehouse → PostHog.** If your source of truth is a database (orders,
  subscriptions), you can sync those rows in as events rather than instrumenting the app.
  **Good for back-office facts** (refunds, fulfilment) that never happen in a browser.
- **PostHog managed Sources.** PostHog's own CDP can pull from Stripe, Hubspot, and
  others directly — useful for joining revenue to behaviour without writing the pipe.

For most teams this course is written for, the answer is: **capture from your own
front-end and back-end, skip the CDP until you have a reason, keep GTM off the product.**

## 3.7 Beating ad blockers

This is the part that surprises people: **10–30% of your events can simply vanish**
because ad blockers maintain blocklists of analytics domains and drop any browser
request to `*.posthog.com`. Your charts under-count, and you don't know by how much.

Two fixes, used together:

### Reverse proxy (recovers client-side events)

Route events through a subdomain of *your own* domain (e.g. `e.yourdomain.com`), which
blockers haven't catalogued. On PostHog Cloud this is a **free, managed** feature:

1. In **Organization settings**, create a managed reverse proxy.
2. Choose a neutral subdomain — *avoid* obvious words like "analytics" or "tracking."
3. Add the CNAME record it gives you; wait a few minutes for SSL.
4. Point your SDK's `api_host` at your subdomain.

This typically recovers 10–30% of otherwise-blocked events. (Note: the managed proxy
runs on Cloudflare and is not HIPAA-compliant — don't route PHI through it. If you self-
host the proxy, prefer a Cloudflare Worker over PostHog's managed CNAME, which can be
"uncloaked" by blocklists over time.)

### Server-side capture (immune by design)

A reverse proxy helps the browser. But the truly robust answer for your *important*
events is to fire them from the **backend** with `posthog-node`. Those events go
server → PostHog and never touch the user's browser, so **ad blockers cannot block
them at all.** This is why §3.3 said capture signup, subscription, and purchase
server-side.

This is also a genuine, defensible positioning point worth understanding: **the events
that drive your lifecycle journeys are server-side and therefore ad-blocker-proof.**
Your client-side product *analytics* still take the ~10–30% hit (only a proxy mitigates
that), but the events that *trigger your emails and Discord messages* are sovereign —
they fire from your server regardless of what's running in the user's browser. Keep
that distinction precise: server-side fixes the *triggers*, a proxy mitigates the
*analytics*.

## 3.8 Events vs Actions

Worth knowing because it saves you re-instrumenting later:

- **Events** are the raw records you capture in code.
- **Actions** are groupings you define *inside PostHog* — combine several events that
  mean the same thing ("all the CTAs that reach /pricing"), or rename a messy
  autocaptured event into something friendly.

The useful part: **actions apply retroactively**, to events you already captured. So if
you realise three different buttons all represent "started checkout," you can unify them
in the UI without shipping code or losing history. It's the escape hatch for the
inevitable day your taxonomy turns out slightly wrong — which is why a *good* taxonomy
matters but a *perfect* one isn't worth blocking on.

## 3.9 Keep it clean

A few habits that prevent the most common messes:

- **Filter out internal/test users** — by email domain, IP, or a test flag — so your
  own activity doesn't pollute the funnels. (Founders refreshing their own app is the
  number-one source of fake activation.)
- **Don't leak PII into event or property *names*** (values are fine; names are not).
- **Version events on breaking changes** (`registration_v2`) rather than silently
  changing a name's meaning.
- **Keep the website and app in one project** so you see the full journey from first
  anonymous visit to paying customer.

### For your kind of startup

- **Mobile-first** — use the native SDK; note that session replay and autocapture behave
  differently than on web, and store-purchase events come from the platform, not a
  browser.
- **B2B** — turn on groups (mind the billing), and track the *account*-level activation,
  not just the user's.
- **E-commerce** — fire `order_completed` server-side from the payment webhook (the
  browser tab often closes before a client event lands), and capture `product_*`
  properties for merchandising analysis.
- **Dev tool** — instrument the docs and the first successful API call; the docs-to-first-
  call path is your real activation funnel.

## 3.10 Do this now

1. Install `posthog-js` with `person_profiles: "identified_only"`.
2. Add `posthog.identify()` at login and `posthog.reset()` at logout.
3. Add the starter events from §3.5 — **fire signup, subscription, and payment from
   your backend.**
4. Write the **tracking plan** (§3.5) — even ten rows in a shared doc.
5. Set up the managed reverse proxy.
6. Decide your one **activation event** and make sure it's firing.

With data flowing, the next chapter is the daily habit that turns all this
instrumentation into decisions.
