Hogsend includes a built-in alerting system that monitors your email and journey metrics and notifies you when thresholds are crossed. Combined with expanded health checks and a dead letter queue, you get full operational visibility without external monitoring tools.

## Alert Rules

Alert rules define conditions that trigger notifications. Each rule monitors a specific metric, fires when a threshold is crossed, and sends a notification through your chosen channel.

### Creating a Rule

```bash
curl -X POST http://localhost:3002/v1/admin/alerts/rules \
  -H "Authorization: Bearer your-api-key" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "High bounce rate",
    "type": "bounce_rate_exceeded",
    "threshold": 0.05,
    "channel": "slack",
    "channelConfig": { "webhookUrl": "https://hooks.slack.com/services/T.../B.../xxx" },
    "cooldownMinutes": 60
  }'
```

### Alert Types

| Type | What it monitors | Threshold meaning |
|---|---|---|
| `bounce_rate_exceeded` | 30-day bounce rate | Rate above this value triggers (e.g., 0.05 = 5%) |
| `journey_failure_spike` | Journey failures per hour | Count above this value triggers |
| `delivery_issue` | Email delivery rate | Rate below this value triggers |
| `high_complaint_rate` | Spam complaint rate | Rate above this value triggers |

### Cooldowns

The `cooldownMinutes` setting prevents alert fatigue. After an alert fires, the same rule will not fire again until the cooldown period has elapsed. Default is 60 minutes.

### Managing Rules

```bash
# List all rules
curl -H "Authorization: Bearer your-api-key" \
  http://localhost:3002/v1/admin/alerts/rules

# Update a rule
curl -X PATCH http://localhost:3002/v1/admin/alerts/rules/rule-uuid \
  -H "Authorization: Bearer your-api-key" \
  -H "Content-Type: application/json" \
  -d '{ "threshold": 0.03, "cooldownMinutes": 120 }'

# Delete a rule
curl -X DELETE http://localhost:3002/v1/admin/alerts/rules/rule-uuid \
  -H "Authorization: Bearer your-api-key"
```

## Notification Channels

Alerts can be delivered through three channels:

### Webhook

Send a POST request to any URL with the alert payload:

```json
{
  "name": "Bounce rate alert",
  "type": "bounce_rate_exceeded",
  "channel": "webhook",
  "channelConfig": {
    "url": "https://your-app.com/webhooks/hogsend-alerts"
  }
}
```

The webhook payload includes the rule name, type, current metric value, threshold, and timestamp.

### Slack

Send to a Slack channel via an incoming webhook:

```json
{
  "name": "Delivery issue",
  "type": "delivery_issue",
  "channel": "slack",
  "channelConfig": {
    "webhookUrl": "https://hooks.slack.com/services/T.../B.../xxx",
    "channel": "#ops-alerts"
  }
}
```

The `channel` field in `channelConfig` is optional -- if omitted, the message goes to the webhook's default channel.

### Email

Send an email notification:

```json
{
  "name": "Complaint rate alert",
  "type": "high_complaint_rate",
  "channel": "email",
  "channelConfig": {
    "to": "ops@yourcompany.com"
  }
}
```

Email alerts are sent via Resend using the same sender configuration as your journey emails.

## Alert History

View past alert triggers to understand patterns and verify notifications are working:

```bash
curl -H "Authorization: Bearer your-api-key" \
  "http://localhost:3002/v1/admin/alerts/history?limit=20"
```

```json
{
  "alerts": [
    {
      "id": "alert-uuid",
      "ruleId": "rule-uuid",
      "ruleName": "High bounce rate",
      "type": "bounce_rate_exceeded",
      "currentValue": 0.062,
      "threshold": 0.05,
      "channel": "slack",
      "delivered": true,
      "triggeredAt": "2025-01-15T08:00:00.000Z"
    }
  ],
  "total": 1,
  "limit": 20,
  "offset": 0
}
```

The `delivered` field indicates whether the notification was successfully sent. If `false`, check the channel configuration (webhook URL reachable, Slack webhook valid, etc.).

## Health Checks

The `GET /v1/health` endpoint reports component-level health for monitoring and Railway health checks.

```bash
curl http://localhost:3002/v1/health
```

```json
{
  "status": "healthy",
  "uptime": 12345.678,
  "timestamp": "2025-01-15T10:30:00.000Z",
  "version": "0.0.1",
  "components": {
    "database": { "status": "up", "latencyMs": 1 },
    "redis": { "status": "up", "latencyMs": 1 }
  }
}
```

### Status Values

| Status | Meaning |
|---|---|
| `healthy` | All components operational |
| `degraded` | One or more components down, but API still serving |

### Component Checks

| Component | What it checks |
|---|---|
| `database` | Executes a simple query against TimescaleDB/Postgres |
| `redis` | Pings the Redis instance |

Each component reports its `status` (`up` or `down`) and `latencyMs` (ping duration in milliseconds). If any component is down, the overall status becomes `degraded` but the API continues to serve requests for features that do not depend on the failed component.

### Monitoring Integration

The health endpoint is designed for:

- **Railway health checks** -- configured in `railway.toml` to restart the service if unhealthy
- **Uptime monitoring** -- external services (Pingdom, Better Uptime, etc.) can poll this endpoint
- **Load balancer health** -- use the HTTP status code (200) for health routing

## Dead Letter Queue

When a task fails after all retry attempts, it is moved to the dead letter queue (DLQ) rather than being silently dropped. The DLQ lets you inspect failures, understand what went wrong, and retry or discard them.

### Viewing the DLQ

```bash
# All pending entries
curl -H "Authorization: Bearer your-api-key" \
  "http://localhost:3002/v1/admin/dlq?status=pending"

# Failed email sends
curl -H "Authorization: Bearer your-api-key" \
  "http://localhost:3002/v1/admin/dlq?source=email&status=pending"
```

```json
{
  "entries": [
    {
      "id": "dlq-uuid",
      "source": "email",
      "sourceId": "email-uuid",
      "payload": {
        "templateKey": "activation/welcome",
        "toEmail": "user@example.com"
      },
      "error": "Resend API timeout after 3 retries",
      "retryCount": 3,
      "status": "pending",
      "retriedAt": null,
      "createdAt": "2025-01-15T10:30:00.000Z"
    }
  ],
  "total": 1,
  "limit": 50,
  "offset": 0
}
```

### DLQ Sources

| Source | Description |
|---|---|
| `email` | Failed email sends (Resend API errors, template rendering failures) |
| `journey` | Journey execution errors that exhausted retries |
| `webhook` | Outbound webhook delivery failures (alert notifications) |

### Retrying a Failed Task

```bash
curl -X POST http://localhost:3002/v1/admin/dlq/dlq-uuid/retry \
  -H "Authorization: Bearer your-api-key"
```

```json
{
  "id": "dlq-uuid",
  "status": "retried",
  "retriedAt": "2025-01-15T11:00:00.000Z"
}
```

The task is re-queued through its original pipeline. If it fails again, it will return to the DLQ with an incremented `retryCount`.

### Discarding an Entry

If a failure is not worth retrying (e.g., the recipient unsubscribed, the event is no longer relevant):

```bash
curl -X DELETE http://localhost:3002/v1/admin/dlq/dlq-uuid \
  -H "Authorization: Bearer your-api-key"
```

```json
{
  "id": "dlq-uuid",
  "status": "discarded"
}
```

Discarded entries remain in the DLQ for audit purposes but are not retried.

## Recommended Setup

A typical production monitoring setup:

1. **Create alert rules** for the metrics you care about most:

```bash
# Alert on bounce rate > 3%
curl -X POST http://localhost:3002/v1/admin/alerts/rules \
  -H "Authorization: Bearer your-api-key" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Bounce rate warning",
    "type": "bounce_rate_exceeded",
    "threshold": 0.03,
    "channel": "slack",
    "channelConfig": { "webhookUrl": "https://hooks.slack.com/services/..." },
    "cooldownMinutes": 120
  }'

# Alert on delivery rate < 95%
curl -X POST http://localhost:3002/v1/admin/alerts/rules \
  -H "Authorization: Bearer your-api-key" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Delivery rate drop",
    "type": "delivery_issue",
    "threshold": 0.95,
    "channel": "slack",
    "channelConfig": { "webhookUrl": "https://hooks.slack.com/services/..." },
    "cooldownMinutes": 60
  }'

# Alert on journey failure spikes (> 10/hour)
curl -X POST http://localhost:3002/v1/admin/alerts/rules \
  -H "Authorization: Bearer your-api-key" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Journey failure spike",
    "type": "journey_failure_spike",
    "threshold": 10,
    "channel": "webhook",
    "channelConfig": { "url": "https://your-app.com/webhooks/alerts" },
    "cooldownMinutes": 30
  }'
```

2. **Set up external uptime monitoring** pointing at `/v1/health`

3. **Review the DLQ weekly** -- check for recurring patterns that indicate systemic issues

4. **Check alert history** to verify notifications are being delivered and thresholds are tuned correctly
