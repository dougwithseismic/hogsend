## Usage

```bash
hogsend init [--force]
```

<Callout type="warn">
`hogsend init` does **not** create or scaffold a project. It **configures an existing Railway deployment** that you created from the Hogsend deploy template. To create a new Hogsend *app* (your own journeys, templates, and config), scaffold it instead:

```bash
pnpm dlx create-hogsend@latest my-app
```

See [Getting started](/docs/getting-started/installation). Today the `@hogsend/*` packages are consumed from this monorepo; the npm publish path is available imminently.
</Callout>

Run `hogsend init` after deploying the [Railway template](https://railway.com/deploy/LxSCyR). The wizard connects to your Railway project, sets environment variables on both the API and Worker services, creates a PostHog webhook destination, redeploys, and verifies the full event pipeline.

## What it does

The init flow has three stages.

### Stage 1: Collect

The wizard prompts for:

1. **Project name** — a human-readable label for this deployment (e.g. "ACME Corp")
2. **Railway API token** — generate one at [railway.com/account/tokens](https://railway.com/account/tokens)
3. **Railway project** — selected from your projects (the one created by the deploy template)
4. **PostHog personal API key** — must start with `phx_`, generated at PostHog > Settings > Personal API Keys
5. **PostHog region** — US (`us.i.posthog.com`), EU (`eu.i.posthog.com`), US (`app.posthog.com`), or a custom self-hosted URL
6. **PostHog project** — selected from your available projects
7. **Resend "from" email** — the sender address for lifecycle emails
8. **Journeys** — multi-select from the CLI's built-in journey list (see the note below)

The CLI authenticates with Railway, discovers the **API** and **Worker** services by name (it looks for `api` and `worker` in the service names), and finds — or generates — the API domain.

### Stage 2: Provision

With your inputs collected, the CLI:

- Generates a `BETTER_AUTH_SECRET` (64-char hex), `POSTHOG_WEBHOOK_SECRET` (32-char hex), and `ADMIN_API_KEY` (32-char hex)
- Sets environment variables on the **API** service: `NODE_ENV`, `PORT`, `LOG_LEVEL`, `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`, `API_PUBLIC_URL`, `POSTHOG_API_KEY`, `POSTHOG_HOST`, `POSTHOG_WEBHOOK_SECRET`, `ADMIN_API_KEY`, `RESEND_FROM_EMAIL`, `ENABLED_JOURNEYS`
- Sets environment variables on the **Worker** service: `NODE_ENV`, `LOG_LEVEL`, `RESEND_FROM_EMAIL`, `ENABLED_JOURNEYS`, `POSTHOG_API_KEY`, `POSTHOG_HOST`
- Creates a PostHog webhook destination pointing to `https://<domain>/v1/webhooks/posthog`, configured with the events required by your selected journeys
- Redeploys both the API and Worker services

### Stage 3: Verify

The CLI waits up to 3 minutes for the API to become healthy, then sends a `hogsend:init_test` event and checks for it in the events store (3 retries at 2-second intervals). If health or verification fails, the config is still saved — you can check later with [`hogsend status`](/docs/cli/status).

## The built-in journey list

<Callout type="info">
The journey multi-select is backed by a **hardcoded 10-journey list baked into the Go CLI**:

`activation-welcome`, `activation-nudge-series`, `conversion-trial-upgrade`, `conversion-abandoned-checkout`, `retention-milestone`, `referral-invite`, `feedback-nps`, `reactivation-dormancy`, `churn-prevention`, `test-onboarding`.

This list (and the trigger events used to build the PostHog webhook) is **decoupled from the journeys your app actually defines**. A scaffolded app owns its own journeys in `src/journeys/` and registers them through `createHogsendClient({ journeys })` / `createWorker({ container, journeys })`. The Go CLI does not read your app's journeys — it only writes the `ENABLED_JOURNEYS` filter. If you author or rename journeys, the engine's filter still applies by ID; the CLI's selector just may not list them.
</Callout>

When all (or none) of the journeys are selected, `ENABLED_JOURNEYS` is set to `*`. Otherwise it is the comma-separated list of selected IDs.

## Flags

| Flag | Description |
|---|---|
| `--force` | Re-run init even if `.hogsend.yaml` already exists |

Without `--force`, running `hogsend init` on a project that is already initialized returns an error with the current project name and slug.

## Output

On success, the CLI saves `.hogsend.yaml` to the current directory and prints a summary card with the API URL, webhook URL, PostHog project link, Railway dashboard link, config file path, and admin key location.

## Requirements

- A Railway project created from the [Hogsend deploy template](https://railway.com/deploy/LxSCyR)
- A Railway API token with access to that project
- A PostHog personal API key (`phx_...`) with project read access
- A verified Resend sending domain
