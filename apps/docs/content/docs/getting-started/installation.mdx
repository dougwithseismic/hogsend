title: Installation
description: Scaffold a fresh Hogsend app, run it locally against Docker, and fire your first journey end-to-end.
---

You install Hogsend by **scaffolding a fresh app** that consumes the versioned `@hogsend/engine` package. You do not clone or fork a monorepo — your repo owns content (journeys, templates, webhook sources, routes, your own migrations) and pins the engine.

## 1. Scaffold the app

```bash
pnpm dlx create-hogsend@latest my-app
# or: npm create hogsend my-app
```

This copies the starter template into `./my-app`, substitutes the app name and pinned engine version, runs `git init` with an initial commit, and installs dependencies. The app name must match `^[a-z0-9][a-z0-9._-]*$`.

Useful flags:

| Flag | Effect |
|---|---|
| `--pm <pnpm\|npm\|yarn\|bun>` | Package manager to use (default `pnpm`) |
| `--no-install` | Skip the dependency install |
| `--no-git` | Skip `git init` and the initial commit |
| `-h`, `--help` | Show help |

The emitted app pins every `@hogsend/*` package to a single engine version line, so the API, worker, and database migrations always move together.

```bash
cd my-app
```

### What you get

The scaffold is a thin app — these are **yours to edit**:

```
my-app/
├─ src/
│  ├─ index.ts              # HTTP entry: createHogsendClient + createApp + boot guard
│  ├─ worker.ts             # Worker entry: createHogsendClient + createWorker
│  ├─ journeys/             # your journeys (welcome.ts, test-onboarding.ts, ...)
│  │  ├─ index.ts           # exports the `journeys` array
│  │  └─ constants/         # your Events / Templates constants
│  ├─ emails/               # your email templates (.tsx) + registry + type augmentation
│  │  ├─ registry.ts        # maps keys → component + subject + category (+ TemplateRegistryMap augmentation)
│  │  └─ welcome.tsx        # your React Email components, yours to edit
│  ├─ webhook-sources/      # your webhook sources (posthog.ts, ...)
│  ├─ workflows/            # extra Hatchet tasks (backfill-example.ts, ...)
│  └─ schema/               # your client-track DB tables
├─ migrations/              # your client-track migrations + ledger
├─ scripts/migrate.ts       # two-track migrate runner
├─ docker-compose.yml       # local Timescale + Redis + Hatchet-Lite
├─ .env.example
├─ tsup.config.ts           # bundles @hogsend/* at build (noExternal)
├─ vitest.config.ts         # inlines @hogsend/engine for tests
└─ package.json             # pins @hogsend/engine + plugins
```

The engine itself is a dependency in `node_modules` — you never edit it. The two example journeys (`welcome`, `test-onboarding`) are emitted into **your** `src/journeys/`; they are yours to keep, change, or delete.

<Callout type="info">
  `@hogsend/*` packages ship **raw `.ts`** (no `dist`). That is why the scaffold sets `tsup.config.ts` `noExternal: ["@hogsend/*"]` (bundles the engine source at build) and `vitest.config.ts` `server.deps.inline: [/@hogsend\/engine/]` (lets Vite transform raw `.ts` for tests). Never run `node src/index.ts` directly — dev is `tsx watch`, production is a `tsup` build then `node dist/...`.
</Callout>

## 2. Configure environment

```bash
cp .env.example .env
```

Fill in the three values you must set:

| Variable | How to get it |
|---|---|
| `BETTER_AUTH_SECRET` | Run `openssl rand -hex 32` (must be ≥ 32 chars) |
| `RESEND_API_KEY` | [Resend dashboard](https://resend.com/api-keys) → API Keys (starts with `re_`) |
| `HATCHET_CLIENT_TOKEN` | The Hatchet dashboard at `http://localhost:8888` after the containers are up (login `admin@example.com` / `Admin123!!`) |

The `.env` lives at your app's root — there is no `apps/api/.env`. The defaults already point `DATABASE_URL` and `REDIS_URL` at the local Docker ports below. See [Configuration](/docs/getting-started/configuration) for every variable.

## 3. Start infrastructure

```bash
docker compose up -d
```

This boots:

- **TimescaleDB** (Postgres 18) on port **5434** — event store, journey state, contacts
- **Redis** 8 on port **6380** — PostHog property caching
- **Hatchet-Lite** — workflow engine; dashboard on **8888**, gRPC on **7077** (with its own internal Postgres)

Open `http://localhost:8888`, log in, and copy the client token into `HATCHET_CLIENT_TOKEN` in your `.env`.

## 4. Run migrations (two tracks)

```bash
pnpm db:migrate
```

A single command runs both tracks in order: the **engine track** first (migrations bundled in `@hogsend/db`, owned upstream), then your **client track** (the `migrations/` folder in your repo). The client track is skipped automatically when it's empty. See [Configuration](/docs/getting-started/configuration#migrations-two-tracks) for details.

<Callout type="warn">
  Use `pnpm db:migrate`, not `pnpm db:push`. `db:push` writes schema objects directly **without** recording a ledger row, which leaves the migration ledger behind the actual schema — the boot guard then refuses to start the API even though the tables exist. Use `db:migrate` from day one.
</Callout>

## 5. Run the app

```bash
# Terminal 1 — HTTP API on http://localhost:3002
pnpm dev

# Terminal 2 — Hatchet worker
pnpm worker:dev
```

`pnpm dev` runs `tsx watch --env-file=.env src/index.ts`; `pnpm worker:dev` runs the same against `src/worker.ts`. The API boot guard checks the **engine** schema and exits if the database is behind (set `SKIP_SCHEMA_CHECK=true` to bypass in an emergency).

Confirm it's healthy:

```bash
curl http://localhost:3002/v1/health
```

You want `"status": "healthy"` with both `schema.engine.inSync` and `schema.client.inSync` `true`. See the [health endpoint reference](/docs/api/ingestion#health) for the full response shape.

## 6. Fire your first journey — same trigger in, different lifecycle

The scaffold ships a `test-onboarding` journey that runs to completion with **no email and no external accounts** — it exists to prove the pipeline end-to-end, and to show off the whole point of code-first lifecycle. Its `run()` branches on `user.properties.plan` and fires internal events with `ctx.trigger` along the way (open `src/journeys/test-onboarding.ts` and read it — it's ~30 lines of plain TypeScript).

Fire its trigger — `test.signup` — **twice**, with a different `plan` each time and a distinct `userId`. With both processes running:

```bash title="pro run"
curl -XPOST http://localhost:3002/v1/ingest \
  -H 'content-type: application/json' \
  -d '{"event":"test.signup","userId":"smoke-pro","userEmail":"pro@example.com","properties":{"plan":"pro"}}'
```

```bash title="free run"
curl -XPOST http://localhost:3002/v1/ingest \
  -H 'content-type: application/json' \
  -d '{"event":"test.signup","userId":"smoke-free","userEmail":"free@example.com","properties":{"plan":"free"}}'
```

`POST /v1/ingest` requires `event` + `userId` (`userEmail` and `properties` are optional). Each call returns `202` and pushes the event to Hatchet, which routes it to the `test-onboarding` journey (its `trigger.event` is `test.signup`).

### Read the divergence

Each run leaves a trail of internal events you can read straight back out of the event store. The admin events feed is **authenticated** — set `ADMIN_API_KEY` in your `.env` (any value) and pass it as a Bearer token. It filters by `userId`, so pull each run's trail separately:

```bash title="pro run's trail"
curl 'http://localhost:3002/v1/admin/events?userId=smoke-pro&limit=10' \
  -H "authorization: Bearer $ADMIN_API_KEY"
```

```bash title="free run's trail"
curl 'http://localhost:3002/v1/admin/events?userId=smoke-free&limit=10' \
  -H "authorization: Bearer $ADMIN_API_KEY"
```

Both runs fire `journey.welcome_fired` then `journey.completed` — but the middle event differs by code path:

- the **pro** run emitted `journey.pro_path`
- the **free** run emitted `journey.free_path`

Same trigger in, **different lifecycle — decided by code you can read** (the `if (user.properties.plan === "pro")` branch in `test-onboarding.ts`), not by a rule buried in a marketing tool. You can also watch each run step through in the [Hatchet dashboard](http://localhost:8888).

<Callout type="info">
  This entire branch needs **zero external accounts** — no Resend, no PostHog, no verified domain. Docker + the two local processes are everything. If the two trails diverged, the engine, worker, ingest pipeline, Hatchet routing, and your journey code are all wired correctly.
</Callout>

## 7. Send a real email (Resend)

The `welcome` example journey (`src/journeys/welcome.ts`) is the one that talks to the outside world. It's triggered by `user.created`: it sends the welcome email **immediately**, then `ctx.sleep(days(2))` — a durable Hatchet sleep — and only then conditionally sends a nudge.

Fire its trigger:

```bash
curl -XPOST http://localhost:3002/v1/ingest \
  -H 'content-type: application/json' \
  -d '{"event":"user.created","userId":"welcome-1","userEmail":"you@yourdomain.com","properties":{"plan":"free"}}'
```

Only the **first** welcome email (subject `Welcome — let's get you set up`) is sent right away. The follow-up nudge sits behind a **2-day durable sleep**, so assert success on the first email only — don't wait around for the second.

<Callout type="warn">
  The one slow prerequisite is a **verified Resend sender**: a real `RESEND_API_KEY` plus a verified domain in the [Resend dashboard](https://resend.com/domains). This is the single setup gate in the whole quickstart — and it sits **off the path** to the aha in step 6. Do step 6 first with zero accounts; only reach for Resend when you want a real email to land.
</Callout>

### Watch it in Studio

With the app running, open Studio:

```text
http://localhost:3002/studio
```

On first run Studio asks you to **create an admin login** — set one, then sign in. Then:

- **[/studio/sends](http://localhost:3002/studio/sends)** — every email the engine sent, with engagement (opened / clicked) as it happens
- **[/studio/journeys](http://localhost:3002/studio/journeys)** — each journey and the contacts moving through it

Your `welcome-1` send shows up under **Sends**; open it and you'll see open and click state populate once you interact with the email.

<Callout type="info">
  Link-click and open tracking is **automatic** — the engine rewrites outgoing HTML to inject tracked links and an open pixel before the provider ever sees it. There is nothing to configure; every send is tracked.
</Callout>

## 8. Did it work? (tiered success)

<Callout type="info">
  **Required — the dependency-free aha (step 6).** You fired `test.signup` twice and read **two different event trails**: the `smoke-pro` run emitted `journey.pro_path`, the `smoke-free` run emitted `journey.free_path`. That alone proves the engine end-to-end with no external accounts.

  **Optional adapters.** Layer these in once the required path is green:
  - **Email** — a real `welcome` send visible in [Studio → Sends](http://localhost:3002/studio/sends), behind a verified Resend sender (step 7).
  - **Analytics** — one PostHog event reaching Hogsend. See [PostHog setup](/docs/getting-started/posthog-setup).
</Callout>

## 9. Author and upgrade

- **Add a journey** — create a file in `src/journeys/` with `defineJourney`, add its constants to `src/journeys/constants/`, and add it to the `journeys` array in `src/journeys/index.ts`. That array is passed to both `createHogsendClient({ journeys })` and `createWorker({ container, journeys })`. Full guide: [Journeys](/docs/guides/journeys).
- **Add an email template** — drop a `.tsx` component in `src/emails/`, register its key in `src/emails/registry.ts` (with the `TemplateRegistryMap` augmentation for type-safe props), and pass the registry as `createHogsendClient({ email: { templates } })`. Full guide: [Email](/docs/guides/email).
- **Upgrade the engine** — bump every `@hogsend/*` package in lockstep, then apply any new migrations:

```bash
pnpm up "@hogsend/*"     # engine + db + core + email + plugins, one version line
pnpm db:migrate          # apply any new engine migrations, then your client track
# verify: GET /v1/health shows engine + client both inSync: true
```

This is never a git merge of a fork. See [Upgrading & Customizing](/docs/operating/upgrading) for the full upgrade flow and the Extend → Patch → Eject ladder — how to change engine behavior without losing clean upgrades.

## Contributing to the engine

The path above is for **building an app on Hogsend**. If you want to develop the engine itself (open a PR against `@hogsend/engine`, a plugin, or the scaffold), you clone the monorepo instead:

```bash
git clone https://github.com/dougwithseismic/hogsend.git
cd hogsend
pnpm install
docker compose up -d
```

The in-repo dogfood app lives at `apps/api` and exercises the engine directly. This is engine-development tooling — it is **not** the client install path. If you're shipping lifecycle journeys for your own product, scaffold an app (steps 1–7); don't clone the monorepo.
