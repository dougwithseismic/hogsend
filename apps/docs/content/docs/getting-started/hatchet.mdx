[Hatchet](https://hatchet.run) is the durable execution engine underneath every journey — it's what makes `ctx.sleep(days(2))` survive a deploy and resume two days later exactly where it left off. Hogsend connects to whatever Hatchet you point it at through three environment variables. Getting a token is the one prerequisite before your first boot.

## The three env vars

The `@hatchet-dev/typescript-sdk` reads these straight from the environment. Hogsend's [env schema](/docs/getting-started/configuration) validates that the token is present and the TLS strategy is one of the allowed values, then hands them to the SDK.

| Variable | Required | Default | What it is |
|---|---|---|---|
| `HATCHET_CLIENT_TOKEN` | **Yes** | — | Bearer JWT for the engine. The SDK throws if it's empty. The only mandatory value. |
| `HATCHET_CLIENT_HOST_PORT` | No | `localhost:7077` | gRPC `host:port`. Falls back to the address embedded in the token, but set it explicitly for clarity. |
| `HATCHET_CLIENT_TLS_STRATEGY` | No | `tls` | One of `none` \| `tls` \| `mtls`. Secure by default (matches Hatchet Cloud); the local hatchet-lite path uses `none`. |
| `HATCHET_CLIENT_NAMESPACE` | No | _(empty)_ | Optional per-tenant isolation namespace on a shared engine. |

The same three values work for every path below — only what you put in them differs. The full annotated env list lives in [`.env.example`](https://github.com/dougwithseismic/hogsend/blob/main/apps/api/.env.example) and on the [Configuration](/docs/getting-started/configuration) page.

## Three ways to get a token

Pick the one that matches how you want to run the engine.

### Path A — self-hosted hatchet-lite (the default)

<Callout type="info">
  For local dev, `pnpm bootstrap` mints this token and writes it to `.env`
  automatically — the manual steps below are for doing it by hand (or when
  bootstrapping a token outside the scaffold flow).
</Callout>

The local `docker compose` (and the [production compose file](/docs/operating/deploy-docker)) runs `hatchet-lite` alongside the app. Since the app needs a token to boot but the token only exists once the engine is up, bring the engine up first, mint a token in its dashboard, then start the app:

```bash
# 1. Bring up just the Hatchet engine.
docker compose up -d hatchet-lite

# 2. Open the dashboard, log in, and mint a token:
#      http://localhost:8888   (default login below)
#      → Settings → API Tokens → create → copy the JWT

# 3. Paste it into .env, then bring up the rest.
#      HATCHET_CLIENT_TOKEN=<the JWT you just copied>
docker compose up -d
```

Default dashboard login: `admin@example.com` / `Admin123!!`.

For hatchet-lite on the compose network the engine speaks insecure gRPC, so:

```bash
HATCHET_CLIENT_HOST_PORT=hatchet-lite:7077
HATCHET_CLIENT_TLS_STRATEGY=none
```

<Callout type="warn">
  **Change the default admin credentials** before exposing the dashboard to any network, and treat the minted token as a secret. The token embeds the engine's gRPC broadcast address, so a token minted against `localhost` won't work from a real domain — if you move the host, mint a fresh token. Set `HATCHET_HOST` to your public host (see the compose file) so the dashboard URL, cookie domain, and the address the token embeds all match what clients dial.
</Callout>

### Path B — Hatchet Cloud (managed)

Zero infrastructure to run. Create a tenant, generate an API token, paste it.

1. Sign up at [Hatchet Cloud](https://cloud.onhatchet.run) and create a tenant.
2. Generate an API token (Settings → API Tokens).
3. Set it:

```bash
HATCHET_CLIENT_TOKEN=<the Cloud token>
HATCHET_CLIENT_HOST_PORT=<your Cloud gRPC address>   # or omit; the token embeds it
HATCHET_CLIENT_TLS_STRATEGY=tls
```

Cloud speaks TLS, so leave `HATCHET_CLIENT_TLS_STRATEGY` at its `tls` default.

### Path C — bring your own Hatchet

Already running a Hatchet engine, or pointing at one your platform team operates? Supply the three values yourself:

```bash
HATCHET_CLIENT_TOKEN=<token from your engine>
HATCHET_CLIENT_HOST_PORT=<your engine's gRPC host:port>
HATCHET_CLIENT_TLS_STRATEGY=<none | tls | mtls — match your engine>
```
