# Walkthrough: `my-first-hogsend`

This folder is a **reference example** of exactly what `create-hogsend` emits when
you run:

```bash
pnpm dlx create-hogsend@latest my-first-hogsend
```

It mirrors the real local scaffolder (`packages/create-hogsend`) output — the
same files a published `--no-install --no-git` run produces (engine pinned at
`0.0.1`), including the one-command `pnpm bootstrap` local setup. **This
directory is not a pnpm workspace member** — it exists purely so you can read
it. Do not run `pnpm install` in the monorepo expecting this to build.

The mental model the scaffold is built around: **the engine is a versioned npm
dependency (`@hogsend/engine` + its sibling `@hogsend/*` packages); your repo is
just content + wiring that you own.** You write journeys, webhook sources,
schema, **your email templates**, and a couple of tiny entry points. The engine
supplies the HTTP server, the worker runtime, the database layer, auth, the email
*provider* + tracking, and all the built-in routes.

## 2. File-by-file

Legend: **OWN** = your content, edit freely. **WIRING** = boilerplate that
plugs your content into the engine; you rarely touch it. **CONFIG** = build/infra
config you tune occasionally.

### Entry points

- **`src/index.ts`** — **WIRING.** The HTTP API process. Builds the Hogsend
  client (`createHogsendClient({ journeys, email: { templates } })`), runs a schema-version
  boot guard, then starts a Hono server via `@hono/node-server`. You almost never
  edit this; the parts you *might* touch are the timeouts and the shutdown list.
  See §4 for what it actually does.
- **`src/worker.ts`** — **WIRING.** The Hatchet worker process. Builds the client
  again (separate process), creates a worker from your journeys, and starts it.
  It also passes `extraWorkflows` (from `src/workflows/index.ts`) so your custom
  Hatchet tasks register out of the box — add yours to that array.

### Email templates — your content

- **`src/emails/`** — **OWN.** Your React Email templates live here, in your repo.
  The engine bakes in no business templates; it owns only the rendering machinery
  and the provider. `welcome.tsx` and `activation-nudge.tsx` are starters keyed
  to `activation/welcome` and `activation/nudge` (the same keys the `welcome`
  journey sends). To change what an email *looks like*, edit the `.tsx` here.
- **`src/emails/registry.ts`** — **OWN.** Maps each template key to its component,
  default subject, category, and optional preview text. It's exported as
  `templates` and passed to `createHogsendClient({ email: { templates } })`, which threads it
  into the engine's tracked mailer at send + render time.
- **`src/emails/templates.d.ts`** — **OWN.** A module augmentation that declares
  each template key and its props on `@hogsend/email`'s open `TemplateRegistryMap`.
  This is what makes `sendEmail({ template, props })` fully type-checked against
  your templates. Add a key here whenever you add one to `registry.ts`.

### Journeys — your core content

- **`src/journeys/index.ts`** — **OWN.** Exports the `journeys[]` array that both
  entry points import. Adding a journey = add it to this array.
- **`src/journeys/welcome.ts`** — **OWN.** The canonical example. Shows the full
  journey anatomy: `meta` (trigger event, `entryLimit`, `suppress` cool-down,
  `exitOn`) plus a `run(user, ctx)` body that sends an email, does a durable
  `ctx.sleep`, checks history, and branches. **Copy this file to make your own
  journeys.** This is the first file most people edit.
- **`src/journeys/test-onboarding.ts`** — **OWN (but really a demo).** A
  trigger-only journey with no email and no external deps. It exists so you can
  fire `test.signup` and watch the whole ingest → Hatchet → journey pipeline run
  end-to-end on a fresh deploy. Delete it once you trust your setup.
- **`src/journeys/constants/index.ts`** — **OWN.** `Events` and `Templates` as
  `as const` objects so journey triggers and email template keys are typed and
  refactor-safe instead of bare strings. The `Templates` keys here line up with
  the keys in `src/emails/registry.ts` (and `templates.d.ts`). Add your event
  names and template keys here.

### Webhook sources

- **`src/webhook-sources/index.ts`** — **OWN.** Exports the `webhookSources[]`
  array passed to `createApp`. These are served at `POST /v1/webhooks/:sourceId`.
- **`src/webhook-sources/posthog.ts`** — **OWN (example).** A `defineWebhookSource`
  that validates a PostHog webhook with Zod, then `transform`s it into an
  ingest event. Copy this to accept Stripe, Clerk, your own backend, etc. The
  `transform` output feeds the same pipeline that drives journeys.

### Workflows (custom Hatchet tasks)

- **`src/workflows/index.ts`** — **OWN.** Exports the `extraWorkflows[]` array of
  your custom Hatchet tasks. `src/worker.ts` imports it and passes it to
  `createWorker`, so anything listed here registers automatically. List only your
  own tasks — the engine's built-ins (`sendEmailTask`, `importContactsTask`,
  `checkAlertsTask`) are registered for you.
- **`src/workflows/backfill-example.ts`** — **OWN (example).** A fully-worked
  batched/idempotent/resumable backfill task for the expand→migrate→contract
  pattern. Wired into `extraWorkflows` by default; remove it (and drop it from
  the array) if you don't need it yet.

### Schema & migrations

- **`src/schema/index.ts`** — **OWN.** Your **client-track** Drizzle tables. Ships
  with a starter `client_notes` table. The header comment is the important bit:
  engine tables (`contacts`, `journey_states`, `email_sends`, tracking, …) live
  in `@hogsend/db` and migrate on a *separate* track — do **not** redefine them
  here.
- **`migrations/`** — **OWN.** Your client-track SQL, generated by
  `pnpm db:generate`. The seeded `0000_init.sql` matches the `client_notes`
  table. The `meta/_journal.json` ledger is what `scripts/migrate.ts` reads to
  decide what to apply.
- **`scripts/migrate.ts`** — **WIRING.** Runs `migrateEngine()` (from
  `@hogsend/db`) first, then your client track. Engine-before-client is
  mandatory so client tables can reference engine tables. This is what
  `pnpm db:migrate` and the Railway pre-deploy command both run.
- **`drizzle.config.ts`** — **CONFIG.** Points drizzle-kit at `src/schema` and the
  `drizzle.__client_migrations` ledger. Critically it uses a *different* ledger
  table from the engine so the two tracks never collide.

### Build / infra / config

- **`scripts/bootstrap.ts`** — **WIRING.** The one-command local setup behind
  `pnpm bootstrap`: checks Docker is running, creates `.env` (with a fresh
  `BETTER_AUTH_SECRET`), auto-remaps any busy host ports, brings up the
  `docker-compose.yml` stack, mints a Hatchet token, and runs both migration
  tracks. Idempotent — safe to re-run any time.
- **`package.json`** — **CONFIG.** Pins all six `@hogsend/*` packages at the
  engine version (`0.0.1`) and defines every script (`bootstrap`, `dev`,
  `worker:dev`, `build`, `db:migrate`, etc.). Upgrading Hogsend = bump these
  pins.
- **`tsup.config.ts`** — **CONFIG.** Bundles (`noExternal`) all `@hogsend/*`
  packages into `dist/` for production, because they ship raw `.ts` with
  `.js`-extension imports that Node can't run directly. npm deps stay external.
- **`vitest.config.ts`** — **CONFIG.** Injects test env vars (so tests need no
  `.env`) and inlines `@hogsend/engine` so Vite can resolve its raw `.ts`.
- **`tsconfig.json` / `biome.json`** — **CONFIG.** Standard strict TS + Biome.
- **`docker-compose.yml`** — **CONFIG.** Local Timescale (5434), Redis (6380), and
  a self-hosted Hatchet-Lite (dashboard 8888, gRPC 7077). `pnpm bootstrap` brings
  it up for you — host ports are parameterized (`${POSTGRES_PORT:-5434}`, …) so it
  can auto-remap any that are already taken. `docker compose up -d` still works if
  you'd rather drive the stack by hand.
- **`railway.toml` / `railway.worker.toml`** — **CONFIG.** Two Railway services
  from one repo: the API (health check + pre-deploy migrate) and the worker (no
  HTTP port). Only relevant when you deploy to Railway.
- **`.env.example`** — **CONFIG.** Every env var the engine reads, with local
  Docker defaults. `pnpm bootstrap` copies it to `.env`, generates
  `BETTER_AUTH_SECRET`, and mints `HATCHET_CLIENT_TOKEN` for you — you only add
  `RESEND_API_KEY` before sending real email.
- **`.node-version` / `.gitignore`** — **CONFIG.** Node 22 pin and standard
  ignores.

## 3. The two processes

Hogsend runs as **two separate Node processes from the same codebase**, exactly
like the main monorepo:

| Process | Entry | Started with | Does |
|---|---|---|---|
| HTTP API | `src/index.ts` | `pnpm dev` / `pnpm start` | serves REST routes, receives ingest events, pushes them to Hatchet |
| Worker | `src/worker.ts` | `pnpm worker:dev` / `pnpm worker` | long-running; executes journey tasks + email sends durably |

Both call `createHogsendClient({ journeys, email: { templates } })` independently. The API
pushes events; Hatchet routes each event to whichever journey task declares it as
a trigger; the worker runs that task.

## 4. What is `createHogsendClient` / `createApp` / `createWorker` actually doing?

These three functions are the entire "engine surface" the scaffold touches.
Here's what each one wires up, in plain language.

### `createHogsendClient({ journeys, email: { templates } })`

Called once per process (both `index.ts` and `worker.ts` call it). It builds one
object — the **client** (a bag of shared services) — that holds every shared
service so nothing has to reach for globals. You hand it two pieces of content:
your `journeys[]` and your `templates` registry. Concretely it constructs and
returns:

- **`env`** — validated environment variables (the process refuses to start if a
  required one like `DATABASE_URL` or `BETTER_AUTH_SECRET` is missing).
- **`logger`** — a Winston logger at your `LOG_LEVEL`.
- **`db`** + **`dbClient`** — the Drizzle database handle and the raw `postgres`
  client (the raw client is what graceful shutdown closes).
- **`auth`** — a configured Better Auth instance (session/user auth), mounted by
  the API at `/api/auth/*`.
- **`email`** — a raw Resend client, plus **`emailService`** — the engine-owned
  **tracked mailer**. It renders your template (from the `templates` registry you
  passed), checks preferences/suppression, rewrites links + injects the open
  pixel, inserts the `email_sends` row, then calls the **provider** to actually
  deliver. The `sendEmail()` you call inside a journey talks to this. Crucially,
  rendering, tracking, DB, and preferences are all engine-owned — only the final
  delivery step is swappable (see `provider` below).
- **`analytics`** — a PostHog capture client, or a no-op if you didn't set
  `POSTHOG_API_KEY`. Supply your own to swap analytics backends.
- **`registry`** — a `JourneyRegistry` built from the `journeys[]` you passed,
  filtered by `ENABLED_JOURNEYS`. This is the index that maps an incoming event
  to the journeys that should fire.
- **`hatchet`** — the Hatchet client used to push events and define tasks.
- **`clientJournal`** — your client migration ledger, used to report client
  schema sync status on `/v1/health`.

Besides `journeys` and `templates`, the factory accepts two more **first-class**
content args:

- **`provider`** — the email provider contract (the swappable "dumb pipe").
  Defaults to a Resend provider built from `RESEND_API_KEY` /
  `RESEND_WEBHOOK_SECRET`. Implement `send(msg)` + webhook parse/verify for
  Postmark/SES/etc. and tracking, rendering, preferences, and the `email_sends`
  pipeline all come along for free, because they never lived in the provider.
- **`analytics`** — your PostHog (or compatible) client; PostHog is the default.

There's also a small **`overrides`** escape hatch (`{ mailer, auth, hatchet,
db }`) for genuinely advanced / test-only swaps — e.g. injecting a mock mailer or
auth in a test. You almost never need it; prefer the first-class
`templates` / `provider` / `analytics` args above. The scaffold doesn't use it.

### `createApp(container, { webhookSources })`

Takes the container and returns the configured Hono app. It wires up the full
middleware stack (secure headers, CORS, compression, request IDs, request
logging, a unified error handler), mounts Better Auth at `/api/auth/*`, exposes
OpenAPI docs at `/docs` + `/openapi.json` (non-production only), and registers
**all the built-in v1 routes** — health, ingest, email
unsubscribe/preferences, admin, tracking (click/open pixels), and the Resend
webhook. Your `webhookSources` are the only content you inject; they become
`POST /v1/webhooks/:sourceId`. In `index.ts` the returned app is handed to
`@hono/node-server`'s `serve()`.

The boot guard above `createApp` is worth understanding: before serving, the API
asks the database what engine schema version it's at. If the DB is *behind* what
this build needs, it logs a clear error and exits rather than 500ing later on a
missing column. (A DB that's *ahead* is fine.) You can bypass with
`SKIP_SCHEMA_CHECK=true`.

### `createWorker({ container, journeys, extraWorkflows })`

Builds the Hatchet worker. It takes your `journeys[]`, filters them by
`ENABLED_JOURNEYS`, turns each into its Hatchet durable task, and registers them
alongside the engine's built-in tasks (`sendEmailTask`, `importContactsTask`,
`checkAlertsTask`). Your `extraWorkflows[]` (from `src/workflows/index.ts`) are
registered too, so custom Hatchet tasks work out of the box. It returns
`{ start, stop }`; `worker.ts` wires `stop()` to SIGTERM/SIGINT for graceful
shutdown, then calls `start()`.

## 5. What lives in the engine vs. your repo

| Lives in `@hogsend/*` (versioned dependency, you don't edit) | Lives in this repo (you own + edit) |
|---|---|
| HTTP server + all v1 routes (health, ingest, admin, email prefs, tracking, Resend webhook) | The `journeys[]` array and every journey file |
| The client factory + all services (db, auth, tracked mailer, PostHog, Hatchet client, logger, registry) | Your email templates (`src/emails/` — `.tsx`, registry, augmentation) |
| The worker runtime + built-in tasks (`sendEmailTask`, `importContactsTask`, `checkAlertsTask`) | The `webhookSources[]` array and each source's `transform` |
| The journey engine: `defineJourney`, enrollment guards, `JourneyContext` (`ctx.sleep`, `ctx.trigger`, `ctx.history`, …) | `Events` / `Templates` constants |
| The tracked mailer (rendering + preferences + link/open tracking + `email_sends`) — provider-agnostic | Your client-track Drizzle schema (`src/schema`) + its migrations |
| The email **provider** contract (Resend by default; swappable for Postmark/SES) | Custom Hatchet tasks in `src/workflows` (wired via `extraWorkflows`) |
| Engine database schema + engine migration track (`contacts`, `journey_states`, `email_sends`, tracking tables) | The two thin entry points (`index.ts`, `worker.ts`) — wiring, but in your repo |
| Condition evaluation, duration helpers (`days`, `hours`, `minutes`) | All config (env, docker, railway, tsup, drizzle, biome) |
| `defineWebhookSource`, the ingestion pipeline | — |

Rule of thumb: **anything in `src/` is yours; anything imported from `@hogsend/…`
is the engine.** You upgrade the engine by bumping the version pins in
`package.json` and running `pnpm db:migrate` for any new engine migrations.

## 6. Honest notes — what's confusing or over-abstracted

A candid read for a newcomer, not a sales pitch:

1. **Two migration tracks is a lot of concept up front.** "Engine track vs.
   client track", two different ledger tables (`__drizzle_migrations` vs
   `__client_migrations`), `scripts/migrate.ts` running both in order, the
   `db:push` ledger gotcha — it's correct and well-documented in the README, but
   it's heavy for someone who just wants to add one table.

2. **The boot guard can feel scary on first run.** `pnpm bootstrap` runs
   `pnpm db:migrate` for you, so the happy path is covered — but if you bring the
   containers up by hand and skip the migration, the API exits with a
   schema-out-of-date error rather than starting. Correct behavior, but a
   beginner may read the hard exit as "the scaffold is broken." The error message
   does tell you to run `db:migrate`, which helps.

3. **`createHogsendClient` is called twice (once per process).** Sensible —
   they're separate processes — but if you assume "the client is one shared
   object for the whole app", the API and worker each having their *own* client
   can surprise you.

4. **`test-onboarding` ships enabled.** Handy for verifying the pipeline, but
   it's a demo journey wired to a real trigger event (`test.signup`). Remember
   to remove it before production so a stray `test.signup` doesn't enroll
   real users.

5. **The `overrides` escape hatch** (`{ mailer, auth, hatchet, db }`) is exposed
   on `createHogsendClient` but the scaffold never uses it. It's a deliberately
   small, advanced/test-only seam — the common config (`templates`, `provider`,
   `analytics`) is first-class. If you stumble onto `overrides` via autocomplete,
   you almost certainly don't want it.

### Does it actually run?

The scaffold is internally consistent and would build/run **once the
`@hogsend/*` packages are published** (the example here was generated with
`--no-install` precisely because those versions aren't on npm yet). With the
packages available, `pnpm install && pnpm bootstrap && pnpm dev` +
`pnpm worker:dev` is a working setup. Custom Hatchet tasks listed in
`src/workflows/index.ts` are wired through `extraWorkflows` and register on
worker start out of the box, and your email templates in `src/emails/` are yours
to edit from day one — no eject required.
